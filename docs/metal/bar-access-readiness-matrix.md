# BAR Access Readiness Matrix

## Purpose

Stage 6 adds a BAR access readiness matrix.

This stage converts the Stage 4 `bar-inventory-summary.json` into a reviewable readiness report.

The expected result is `NOT_READY`.

That is intentional.

The goal is to make sure the project does not silently escalate from read-only BAR inventory diagnostics into PCI config-space writes, MMIO access, BAR memory mapping, GPU reset logic, firmware loading, display initialization, framebuffer initialization, Metal acceleration, or DriverKit activation.

## Inputs

Required input:

    bar-inventory-summary.json

Usually generated by:

    ./scripts/summarize-bar-inventory.py ~/Desktop/H1mekaRTX-bar-inventory-test

## Usage

Generate readiness output:

    ./scripts/generate-bar-readiness-matrix.py ~/Desktop/H1mekaRTX-bar-inventory-test/bar-inventory-summary.json

Optional output directory:

    ./scripts/generate-bar-readiness-matrix.py ~/Desktop/H1mekaRTX-bar-inventory-test/bar-inventory-summary.json --out-dir ~/Desktop/H1mekaRTX-bar-inventory-test

## Outputs

The script writes:

- `bar-access-readiness-matrix.json`
- `bar-access-readiness-matrix.md`

## Readiness States

| State | Meaning |
| --- | --- |
| `INVALID` | The input summary failed required validation. |
| `NOT_READY` | The input is valid, but access is still blocked by unresolved research and safety requirements. |
| `READY_FOR_REVIEW` | Reserved for a future stage after all blockers are resolved. |

## Expected Stage 6 Result

Expected result:

    NOT_READY

This is not a failure.

It means the safety gates are working.

## Current Blocking Areas

- BAR identity
- BAR size semantics
- register map
- register side effects
- recovery path
- DriverKit activation design
- Metal acceleration boundary

## Safety Boundary

This stage is read-only.

It only reads an existing Stage 4 JSON file and writes local report files.

It does not perform:

- ioreg collection
- system_profiler collection
- PCI config-space reads
- PCI config-space writes
- MMIO reads
- MMIO writes
- BAR memory mapping
- BAR memory poking
- GPU reset logic
- firmware loading
- GSP initialization
- display engine initialization
- framebuffer initialization
- Metal acceleration attempts
- DriverKit activation

## Notes

This stage intentionally blocks escalation.

Future BAR/MMIO work should be split into smaller reviewable stages, and each stage should preserve a clear read-only boundary unless explicitly reviewed otherwise.
